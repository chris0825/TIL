|헤|2|Host: 상대방(서버) 주소|

### 방식필드 종류
- GET
  > 브라우저가 URL 필드로 식별되는 객체를 요청할 때 사용
- 조건부 GET
  > GET 방식 + If-Modified-Since<br>
  > If-Modified-Since == Last-Modified 일때 캐시 사용 (!=면 일반 GET처럼 받아오고 캐시에 저장)
- POST
  > 사용자(클라이언트)의 입력이 필요할 때<br>
  > ex) 회원가입, 장바구니
- HEAD
  > 웹 서버 정보만 필요할 때 (바디문 전송 X)<br>
  > ex) 디버깅
- PUT
  > 웹 서버에 업로드 할 때<br>
  > ex) 과제, 파일 업로드
- DELETE
  > 웹 서버에서 PUT으로 올린 객체를 삭제
### HTTP 응답 메세지
> 상태 라인: 프로토콜 '버전필드', '상태코드', '해당 상태 메세지'
- 상태코드 종류
  - 200 OK
    > 요청 성공
  - 301 Moved Permanently
    > URL이 변경되어 바뀐 URL로 안내
  - 304 Not modified
    > 바뀐게 없으므로 캐싱 복사복을 이용
  - 400 Bad Request
    > 서버가 클라이언트의 요청 메세지를 이해하지 못함
  - 404 Not Found
    > 요청 문서가 서버에 존재하지 않음
  - 505 HTTP Version Not Supported
    > 요청 HTTP 프로토콜 버전을 지원하지 않음
- 헤더라인
  - connection
    > 클라이언트에게 메세지를 보낸 후 TCP 연결을 열고 닫음
  - Date
    > HTTP 응답이 서버에 의해 생성되고 보낸 날짜와 시간을 나타냄
  - Server
    > 메세지가 생성된 웹 서버 정보
  - Last-Modified
    > 객체가 생성되거나 마지막으로 수정된 시기
  - Content-Length
    > 송신되는 객체의 바이트 수
  - Content-Type
    > 개체 몸체 내부의 타입
- 개체 몸체
### 웹 캐시 (= 프록시 서버)
> 웹 서버를 대신하여 HTTP 요구를 충족시키는 네트워크 개체
### HOL 블로킹
> 하나의 처리가 오래걸려 뒤에 대기중인 작은 객체들의 처리가 밀림(병목)
### QUIC
> 트랜스포트 프로토콜 <br>
> UDP 프로토콜 위에 위치하는 애플리케이션 계층에서 구현
### 인터넷 메일 시스템
- 사용자 에이전트
  > 사용자가 메세지를 읽고, 응답하고, 전달하고, 저장하고, 구성하게 해줌 <br>
  > ex) outlook, mail 앱 등
- 메일 서버
  > 메일 박스를 가짐 <br>
  > 수신자의 메일 박스가 고장나면 송신자는 메세지 큐에 보관하고 약 30분마다 재전송을 시도함. 그래도 몇일동안 보내지 못하면 송신자에게 통보
- SMTP
  > 인터넷 전자메일을 위한 주요 애플리케이션 계층 프로토콜 <br>
  > RFC 5321에 정의됨 <br>
  > PUSH 프로토콜 <br>
  > 두 메일 서버(송/수신)간 거리가 멀어도 중간서버 없이 direct로 전송 <br>
  > 지속 연결 사용 <br>
  > TCP의 신뢰적인 데이터 전송 서비스에 의존
- IMAP (인터넷 메일 접근 프로토콜)
  > PULL 프로토콜
### 호스트 식별자
- 호스트 이름
  > **장:** 기억이 쉬움 <br>
  > **단:** 위치에 대한 정보 얻기 힘듦
- IP주소
  > **특징:** 계층구조(왼쪽에서 오른쪽으로 조사) <br>
  > 위치에 대한 정보를 얻을 수 있음
### DNS
> 호스트 이름을 IP 조소로 변환 <br>
> UDP 상에서 수행됨 <br>
> Port NO. 53 <br>
> 호스트 에일리어싱 (<> 정식 호스트 이름) <br>
> 메일 서버 에일리어싱 <br>
> 부하 분산(중복 서버) <br>
> 중앙 집중 방식은 오늘날 적합 X (서버의 고장, 트래픽 양, 먼 거리, 유지관리, 확장성)
- DNS 계층
  - 루트 DNS 서버
    > TLD에서 IP 주소를 제공
  - 최상위 레벨 도메인 서버 (TLD)
    > 책임 DNS 서버에 대한 IP 주소를 제공<br>
    > 국가급 (.kr, .uk, .jp  등)<br>
    > 성질 (.com, .edu 등)
  - 책임 DNS 서버
    > 기관급 (facebook, instagram 등)
- 로컬 DNS 서버
  > 계층구조에 해당하지 X<br>
  > 캐시 역할로 루트 DNS 부하를 낮추는 역할
### DNS 캐싱
> 메세지 수 감소 => 지연 성능의 향상
### 자원 레코드 (RR)
> DNS 서버가 호스트 이름을 IP주소로 매핑하기 위해 필요<br>
> Name, Value, Type, TTL(Time To Live)
- type = A
  > Name = 호스트 이름, Value = 호스트 이름에 대한 IP주소
- type = NS
  > Name = 도메인, Value = 도메인 내부의 책임 DNS서버의 호스트 이름
- type = CNAME
  > Name = 별칭 호스트 이름, Value = 정식 호스트 이름
- type = MX
  > Name = 별칭 호스트 이름, Value = 메일 서버의 정식 이름

    - 책임 플래그
    - 재귀 요구 플래그
    - 재귀 가능 플래그
- 질문 영역
- 답변 영역
- 책임 영역
- 추가 영역

### 등록 기관

### DNS 취약점
- DDOS
  > root DNS, TLD서버에 대량의 질의를 전송시켜 마비
- 중간자 공격
  > 호스트가 보내는 질의를 중간에 가로채 다른 응답을 반환
- DNS 중독
  > DNS 서버로 가짜 레코드를 보내 해당 DNS 캐시에 가짜 캐시 레코드를 등록시킴
### ➡️ DNSSEC(무결성, 암호화)를 통해 해결 가능
### P2P 특징
> 서버가 항상 켜져있지 X<br>
> 종단 시스템끼리 서로 서버<>클라이언트 역할 바꿔가며 수행<br>
> 종단 시스템이 동적 IP를 이용하기 때문에 관리가 어려움<br>
> 한 종단 시스템으로 부터 자료를 받는 것이 아니라 여러 종단 시스템과 연결하여 부분적으로 번갈아가면서 수행함

  > 실제로 통신하고 있는 피어

### 클러스터 선택 정책
> 클라이언트를 동적으로 어떤 서버 클러스터 또는 CDN 데이터 센터로 연결<br>
> 지리적으로 가장 가까운 클러스터 할당 기법<br>
> 현재의 네트워크의 트래픽 상황을 반영한 최선의 클러스터를 선택하기 위해 CDN은 주기적으로 클러스터와 클라이언트 간 지연 및 성능에 대해 실시간으로 측정
### 아마존 클라우드의 예시
- 콘텐츠 수집
- 콘텐츠 처리
- CDN으로 버전 업로드
### 트랜스포트 계층(layer 4) 프로토콜은 각기 다른 호스트에서 동작하는 애플리케이션 프로세스간 논리적 통신을 제공
> 논리적 통신 = 애플리케이션의 관점에서 보면 동작하는 프로세스의 호스트들이 서로 직접 연결된 것 처럼 보임
### IP
> 네트워크 계층(layer 3) 프로토콜
- 최선형 전달 서비스
  > IP가 통신하는 호스트간 세그먼트를 전달하기 위해 최대한 노력하지만 보장 X<br>
  > 내부 데이터 무결성 보장 X<br>
  > ➡️ 비신뢰적 서비스
### 각 호스트는 적어도 하나의 IP주소를 보유
### 트랜스포트 계층의 다중화, 역다중화
- 다중화
  > 출발지 호스트 소켓으로부터 데이터를 모으고, 이에 대한 세그먼트를 생성하기 위해 각 데이터의 헤더 정보를 캡슐화 하고 그 세그먼트들을 네트워크 계층으로 전달하는 작업
- 역다중화
  > 트랜스포트 계층 세그먼트의 데이터를 올바른 소켓으로 전달하는 작업
### TCP
- 신뢰적인 데이터 전송
  - 흐름 제어
  - 순서번호
  - 확인 응답(ACK/NAK)
  - 카운트다운 타이머
- 혼잡 제어
### 세그먼트는 전달될 적절한 소켓을 가리키는 특별한 필드(출발지, 목적지 포트번호 필드)를 갖는다.
### 포트번호 0~1023은 잘 알려진 포트번호로 취급되며 사용이 엄격하다.
### UDP
- 비연결형
  > 송신 개체와 수신 개체간 핸드셰이크(연결 설정이) 없음
- 애플리케이션 레벨 정교한 제어가 가능
- 연결 상태가 없음
  > 연결 상태 유지X, 기록X
- 작은 패킷 헤더 오버헤드
  > TCP 헤더가 20바이트인 반면, UDP 헤더는 8바이트임
### UDP 체크섬
1. 검사하고자 하는 비트의 총 합을 구함 (이때, 자리넘침은 버림 (= 윤회식 자리올림))
2. 1에서 나온 결과를 1의 보수 취함 (체크섬 비트)
3. 1의 합과 2를 더했을 때, 모든 비트가 1이면 오류가 없는 것.
### 종단과 종단의 원칙
> 어떤 기능이 종단 기반으로 구현돼야 하므로 하위 레벨에 있는 기능은 상위 레벨에서 이들을 제공하는 비용과 비교했을 때 중복되거나 거의 가치가 없을 수 있다.
