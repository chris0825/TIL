# 소프트웨어 테스트
## 애플리케이션 테스트 기본 원리
  > 테스트는 기본적으로 결함이 존재함을 밝히는 것(무결함을 증명할 수 X)
  
  -> 완벽한 테스트는 불가능(∵ 무한 경로, 무한 입력 불가)
  - **결함 집중** - 파레토(Pareto) 법칙
    > 20% 모듈에서 전체 결함의 80%가 발생
  - **살충제 패러독스**
    > 동일한 테스트 케이스에 의한 반복 테스트는 새로운 버그를 발견하지 못한다
  - **오류-부재의 궤변**
    > 결함이 없다해도 사용자의 요구사항을 충족하지 못할 시 품질이 저하된다
  - **Brooks의 법칙**
    > 지연되는 프로젝트에 인력을 추가 투입하면 프로젝트는 더 지연된다
## 애플리케이션 테스트 분류
### 프로그램 실행 여부
- 정적 테스트
  > 프로그램을 실행하지 않고 명세서와 소스코드만 분석<br>
  > Ex) 동료 검토, 워크 스루, 인스펙션, 코드 검사
- 동적 테스트
  > 프로그램 실행 후 오류 검사<br>
  > Ex) 화이트/블랙박스 테스트
### 테스트 기반 테스트
- **명세 기반**
  > 사용자의 요구 사항에 대한 명세를 빠짐 없이 테스트 케이스로 구현하는지 확인<br>
  > Ex) 동등 분할 / 경계값 분석 (블랙박스 테스트)
- **구조 기반**
  > SW 내부 논리 흐름에 따라 테스트 케이스 작성 및 확인<br>
  > Ex) 구문 기반 / 결정 기반 / 조건 기반 (화이트박스 테스트)
- **경험 기반**
  > 테스터의 경험을 기반으로 수행<br>
  > Ex) 에러 추정, 체크리스트, 탐색적 테스팅
### 목적 기반 테스트
- **회복(Recovery)**
  > 시스템에 인위적 결함을 부여한 후 프로그램이 정상적으로 회복되는 과정을 확인
- **안전(Security)**
  > 외부 불법 침입으로부터 시스템을 보호할 수 있는지 확인
- **강도(Stress)**
  > 과부하시 SW가 정상적으로 구동하는지 여부를 확인
- **성능(Performance)**
  > 실시간 성능 및 전체적인 효율성을 진단(응답 시간, 업무 처리량)
- **구조(Structure)**
  > SW 내부 논리적 경로 및 소스 코드 복잡도를 평가
- **회귀(Regression)**
  > SW 내 변경 또는 수정된 코드에 새로운 결함이 없는지 확인
- **병행(Parallel)**
  > 변경 및 기존 SW에 동일한 데이터를 입력 후 결과를 비교
### 시각(관점) 기반 테스트
- **검증(Verification)**
  > 개발자의 시각에서 제품의 생산 과정 테스트<br>
  > Ex) 단위/통합/시스템 테스트
- **확인(Validation)**
  > 사용자의 시각에서 생상된 제품의 결과 테스트<br>
  > Ex) 인수 테스트(알파 / 베타)
## 화이트박스 테스트(White Box Test)
  > 모듈 안의 내용(작동 과정)을 직접 볼 수 있으며, 내부의 논리적인 모든 경로를 테스트
  > 소스 코드의 모든 문장을 한 번 이상 수행하여 논리적 경로를 점검(선택, 반복 수행)
  > 테스트 데이터를 선택하지 위해 검증 기준 커버리지를 정함
### 화이트박스 테스트 검증 기준
- **구문 커버리지(Statement Coverage)**
  > 프로그램 내 모든 명령문이 적어도 한 번 수행
- **결정(분기) 커버리지(Branch Coverage)**
  > 프로그램 내 전체 결정문이 적어도 한 번은 참/거짓 결과를 수행
- **조건 커버리지(Condition Coverage)**
  > 결정 명령문 내의 각 개별 조건식이 적어도 한 번은 참/거짓 결과를 수행
- **조건/결정 커버리지(Condition/Decision Coverage)**
  > 전체 조건식 뿐만 아니라 개별 조건식도 참과 거짓을 각각 한 번 이상 결과를 수행
- **변경 조건/결정 커버리지(MC/DC Coverage)**
  > 각 개별 조건식이 독립적으로 전체 조건식의 결과에 영향
- **다중 조건 커버리지(Multiple Condition Coverage)**
  > 결정 포인트내 모든 개별 조건식의 모든 가능한 논리적 조합을 고려하여 100% 커버리지를 보장
### 화이트박스 테스트 종류
- **기초 경로 검사(Base Path Testing)**
  > 대표적인 화이트박스 테스트 기법(동적 테스트)<br>
  > 테스트 케이스 설계자가 절차적 설계의 논리적 복잡성을 측정할 수 있게 해주는 테스트 기법<br>
  > 측정 결과는 실행 경로의 기초를 정의하는 지침으로 사용
- **제어 구조 검사(Control Structure Testing)**
  > 조건 검사: 프로그램 모듈 내 논리적 조건 테스트<br>
  > 루프 검사: 프로그램 반복(Loop) 구조 테스트<br>
  > 자료 흐름 검사: 변수의 정의와 변수 사용의 위치 테스트
## 블랙박스 테스트(Black Box Test)
  > 모듈 내부의 내용을 알 수 없으며, 소프트웨어 인터페이스에서 실시되는 테스트<br>
  > SW 각 기능이 완전히 작동되는 것을 입증하는 테스트로 **기능 테스트**라고 함
### 블랙박스 테스트 종류
- **동치 분할 검사(Equivalence Partition)**
  > 프로그램 입력 조건에 타당한 입력 자료와 타당하지 않은 입력 자료의 개수를 균등하게 진행
- **경계값 분석(Boundary Value)**
  > 입력 조건의 경계값으 테스트 케이스로 선정<br>Ex) 범위 구간의 양 끝: ~이상/이하/초과/미만
- **원인-효과 그래프 검사(Cause-Effect Graphing)**
  > 입력 데이터 간의 관계와 출력에 영향을 미치는 상황을 체계적으로 분석 후 효용성이 높은 테스트 케이스를 선정
- **오류 예측 검사(Error Guessing)**
  > 과거의 경험이나 확인자의 감각으로 테스트를 진행
- **비교 검사(Comparison)**
  > 여러 버전의 프로그램에 동일한 결과가 출력되는지를 확인
## 요구사항 검증(Requirements Verification)
  > 실제로 고객이 원하는 시스템을 제대로 정의했는지 점검하는 과정(실제 요구사항이 반영되었는지 여부와 문서상 요구사항이 서로 상충되지는 않는지 점검)<br>
  > 시스템 개발 완료 후 문제 발생시 막대한 재작업 비용이 발생되기에 검증이 중요함
  - **동료 검토(Peer review)**
    : 작성자가 내용 설명 후 동료들이 결함을 검토
  - **워크 스루(Walk through)**
    : 요구사항 명세서를 미리 배포 후 짧은 검토 회의를 진행
  - **인스펙션(Inspection)**
    : 작성자를 제외한 다른 전문가들이 결함을 검토
### 검증시 체크리스트
- 기능성(Functionality)
- 완전성(Completeness)
- 일관성(Consistency)
- 명확성(Unambiguity)
- 검증 가능성(Verifiability)
- 추적 가능성(Traceability)
- 변경 용이성(Easily Changeable)
## 개발단계에 따른 애플리케이션 테스트
### 단위 테스트(Unit test)
  > 최소 단위(모듈<sub>: 소프트웨어를 이루는 기본 단위(독립적 기능)</sub>/컴포넌트) 기반 테스트<br>
  > 주로 구조 기반 테스트 진행 / 기능성 테스트 최우선
### 통합 테스트(Integration test)<sub>: 인터페이스 간 시스템이 정상적으로 실행되는지 확인</sub>
  > 단위 테스트 후 모듈을 통합하는 과정에서 발생하는 오류 및 결함을 찾는 테스트 기법
  - **하향식(Top-down)**
    > 상위 모듈에서 하위 모듈 방향으로 통합<br>
    > 깊이 우선(Depth first) 통합법 / 넓이 우선(Breadth first) 통합법<br>
    > 초기부터 사용자에게 시스템 구조를 보여줌<br>
    > 스텁(Stub): 모듈의 기능을 단순히 수행하는 도구(시험용 모듈)
  - **상향식(Bottom-up)**
    > 하위 모듈에서 상위 모듈 방향으로 통합<br>
    > 하나의 주요 제어 모듈과 관련된 종속 모듈의 그룹인 클러스터(Cluster)와 드라이버(Driver)사용<br>
    > 스텁 미사용
### 시스템 테스트(System test)
  > 개발된 SW의 컴퓨터 시스템 내 작용 여부를 점검<br>
  > 실제 사용 환경과 유사한 환경에서 기능적(블랙박스) 및 비기능적(화이트박스) 테스트
### 인수 테스트(Acceptance test)
  > 사용의 요구 사항을 충족하는지 확인
  - **알파 테스트**
    > 통제된 환경에서 사용자가 개발자와 함께 확인
  - **베타 테스트**
    > 통제되지 않은 환경에서 개발자 없이 여러 명의 사용자가 검증
### 테스트 관련 기타 용어
- **테스트 시나리오(Test Scenario)**
  > 테스트 케이스 적용/동작 순서에 따라 여러 테스트 케이스를 묶은 집합
- **테스트 오라클(Test Oracle)**
  > 테스트 결과의 참/거짓 판단을 위해 사전에 정의된 참 값을 대입
  - **참 오라클(True)**
    > 모든 테스트 케이스 입력값에 기대 결과 제공
  - **샘플링 오라클(Sampling)**
    > 특정 테스트 케이스 입력값에 기대 결과 제공
  - **추정 오라클(Heuristic)**
    > 특정 테스트 케이스 입력값에 기대 결과를 제공하고<br>
    > 나머지 테스트 케이스 입력값에는 추정 결과를 제공
  - **일관성 오라클(Consistent)**
    > 테스트 케이스의 수행 전/후의 결과값이 동일한지 확인
## 테스트 하네스(Test Harness)
### 테스트 드라이버(Test Driver)
  > 시험 대상의 하위 모듈을 호출하여 모듈 테스트 수행 후의 결과를 도출함<br>
  > -> 상향식 통합 테스트에서 사용
### 테스트 스텁(Test Stub)
  > 제어 모듈이 호출하는 하위 모듈의 역할 단순 수행<br>
  > -> 하향식 테스트에서 사용
### 테스트 슈트(Test Suites)
  > 시스템에 사용되는 테스트 케이스의 집합(컴포넌트 / 모듈)
### 테스트 케이스(Test Case)
  > 사용자의 요구사항을 준수했는지 확인하기 위해 설계된 테스트 항목 명세서(입력 값, 실행 조건, 기대 결과 등)
### 테스트 스크립트(Test Script)
  > 자동화된 테스트 실행 절차에 대한 명세서
### 목 오브젝트(Mock Object)
  > 사용자의 행위를 예측(추정)한 값을 입력 시 계획된 행위를 수행하는 객체
